# Making an LLM-Agent Workflow System Actually Get Followed

**Research report — 2026-07-05**
**Question:** How do you make a skill/workflow system for LLM coding agents *reliably followed* (behavioral compliance/enforcement), and why did the `obra/superpowers` approach work where embedding the same processes into agent prompts did not?

---

## Executive summary (~200 words)

The core finding is a two-layer model of compliance: **structure (soft) plus interception (hard)**, and the failure mode this repo hit is a predictable one — *embedding a process as reference prose deletes the single most important mechanism, the discrete invoked action*.

Superpowers works not because its prose is persuasive but because it converts "follow this process" into a **sequence of atomic, announced, invoked steps** gated by an always-on rule ("if there is even a 1% chance a skill applies, you MUST invoke it — you do not have a choice"). Compliance comes from four reinforcing forces: (1) an always-on **gateway** injected every turn that forces a skill check *before any response*; (2) **just-in-time loading** — the heavy procedure enters context only when invoked, dodging "lost-in-the-middle"/context-rot degradation; (3) **commitment/announcement** — the agent must declare the skill it is using, which research and Anthropic's own guidance associate with higher adherence; and (4) **pressure-tested prose forms** (Iron Law + rationalization tables + red flags) empirically hardened against the exact excuses agents generate.

Embedding those same processes as always-present system-prompt prose loses (1), (2), and (3) at once: nothing forces invocation, the guidance sits in the low-attention middle of a long prompt, and there is no commitment step. The agent reads it and drifts.

The highest-leverage fixes, in order: restore an **always-on gateway skill/injection**; keep procedures as **discrete invokable skills**, not embedded prose; add **hard gates** (OpenCode `permission` maps and, where available, `tool.execute.before` deny) so violations are *blocked*, not merely discouraged; and **pressure-test** the wording.

**Two OpenCode-specific cautions verified below:** the `experimental.chat.system.transform` mutation path was reported broken and closed "not planned" (#17100), and plugin `tool.execute.before` hooks do **not** intercept subagent tool calls (#5894) — so hard enforcement on OpenCode should lean on native `permission` maps and a `messages.transform`-based (not `system.transform`-based) injection, not on those two paths.

## 1. The obra/superpowers design philosophy

**Primary sources:** the repo [github.com/obra/superpowers](https://github.com/obra/superpowers); Jesse Vincent's ("obra") blog [Superpowers: How I'm using coding agents in October 2025](https://blog.fsck.com/2025/10/09/superpowers/); the [`writing-skills/SKILL.md`](https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md); DeepWiki's writeups of [`using-superpowers`](https://deepwiki.com/obra/superpowers/7.1-using-superpowers-(meta-skill)) and [TDD-for-skills](https://deepwiki.com/obra/superpowers/8.1-test-driven-development-for-skills); and a third-party analysis, [Superpowers: the technology to "persuade" AI agents](https://dev.to/tumf/superpowers-the-technology-to-persuade-ai-agents-why-psychological-principles-change-code-quality-2d2f).

Superpowers is "a plugin, a container for skills, slash commands, hooks, subagents, and a session-start bootstrap," where the actual capabilities are individual `SKILL.md` files ([blog.fsck.com](https://blog.fsck.com/2025/10/09/superpowers/)). It ships a `brainstorm → plan → implement` workflow that Claude enters *by default* when it detects you're starting a task, without a slash command. Each structural element has an explicit compliance rationale:

**The `using-superpowers` always-on gateway (the load-bearing element).** Unlike every other skill, `using-superpowers` is *injected into session context at startup* via a `SessionStart` hook, making it "the constitutional layer of the system" ([DeepWiki](https://deepwiki.com/obra/superpowers/7.1-using-superpowers-(meta-skill))). It carries the **Iron Law / 1% rule**: *"If there is even a 1% chance a skill might apply, you must invoke it,"* and *"IF A SKILL APPLIES TO YOUR TASK, YOU DO NOT HAVE A CHOICE. YOU MUST USE IT. This is not negotiable. This is not optional."* Crucially it fires **before any response** — *"Invoke relevant or requested skills BEFORE any response or action"* — so even a clarifying question must be preceded by a skill check. *Theory of why it works:* it removes the decision point at which an agent would rationalize a shortcut, and because it is re-injected every turn it never falls out of attention (see §2).

**The `<EXTREMELY-IMPORTANT>` / "you do not have a choice" pressure language.** The bootstrap uses caps-locked urgency ("EXTREMELY_IMPORTANT", "RIGHT NOW, go read") to frame skills as constraints, not suggestions ([blog.fsck.com](https://blog.fsck.com/2025/10/09/superpowers/)). Vincent explicitly frames this via Cialdini's *Influence* — authority, commitment, liking, reciprocity, scarcity, social proof, unity — and quotes Claude's own assessment that these are used "not to jailbreak me, but to make me MORE reliable and disciplined." *Theory:* absolute/imperative framing leaves no linguistic room to negotiate; soft framing ("prefer…", "consider…") reopens negotiation (confirmed by the skill-writing guidance below).

**"Use when…" descriptions.** Descriptions must *"Start with 'Use when…' focusing on triggering conditions only"* and must **never summarize the workflow** — the guidance warns a workflow summary "creates shortcuts agents will take" ([writing-skills/SKILL.md](https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md)). *Theory:* the description is read at *discovery* time to decide whether to load the skill; if it leaks the procedure, the agent acts on the summary and never loads the full, hardened body. Keep the trigger; hide the recipe.

**Iron Law framing + rationalization tables + red-flag lists.** "Each rigid skill opens with a capitalized, non-negotiable rule, an Iron Law, followed by a table of red flags: the rationalizations the agent is most likely to use to skip the rule" ([Marc Nuri](https://blog.marcnuri.com/superpowers-claude-code-skills-framework)). Example (TDD): *"NO SKILL WITHOUT A FAILING TEST FIRST … Write skill before testing? Delete it. Start over."* The **rationalization table** pairs each anticipated excuse with a rebuttal ("Too simple to test" → "Simple code breaks; testing takes 30 seconds"); the **red-flags list** is a set of self-check triggers ("Code before test", "I already manually tested it", "This is different because…") that all resolve to a single action. *Theory:* the agent's own most-likely evasions are pre-empted in-context, so when it starts to generate the excuse it meets the counter it was about to rationalize past.

**Graphviz decision flowcharts.** Prescribed *only* for "non-obvious decision points, process loops where you might stop too early, and 'when to use A vs B' decisions" — never for reference, code, or linear instructions ([writing-skills/SKILL.md](https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md)). *Theory:* a flowchart forces an explicit branch selection at exactly the points where drift ("I'll just stop here") occurs.

**Checklists / todos.** Skills mandate *"Create a todo for EACH checklist item below."* *Theory:* decomposition into externally tracked items converts a vague obligation into discrete, individually verifiable states, and a visible open todo resists being silently skipped.

**"Match the form to the failure" (the deepest insight).** The skill-writing guidance gives a table mapping failure type → correct prose form, with *why the wrong form backfires*: rule-violation-under-pressure needs *prohibition + rationalization table + red flags* (soft "prefer…" "reopens negotiation"); a wrong output shape needs a *positive recipe* (prohibitions "create competing interpretations"); a missing element needs a *structural template with REQUIRED fields* ("prose reminders = agents skip them"); conditional behavior needs an *observable predicate* ("if X exists, do Y") because "unconditional + exemption clauses = loopholes." A hard constraint: **"No nuance clauses"** — appending "unless it matters" reopens negotiation. This is the single most transferable lesson about *wording* for compliance.

**Pressure-scenario TESTS for skills (TDD-for-prompts).** The **Iron Law of skill authoring** is *"NO SKILL WITHOUT A FAILING TEST FIRST."* The RED-GREEN-REFACTOR cycle: RED = run pressure scenarios *without* the skill and capture the exact verbatim rationalizations a fresh agent produces (with a no-guidance control to prove the failure exists); GREEN = write the minimal skill countering those specific failures; REFACTOR = find the new rationalizations that emerge and add explicit counters, re-testing 5+ repetitions per variant ([writing-skills/SKILL.md](https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md), [DeepWiki TDD-for-skills](https://deepwiki.com/obra/superpowers/8.1-test-driven-development-for-skills)). Scenarios weaponize Cialdini pressures — time ("Every minute costs $5k"), sunk cost ("You just spent 45 minutes writing async test infrastructure. It works"), authority, exhaustion ([blog.fsck.com](https://blog.fsck.com/2025/10/09/superpowers/)). *Theory:* the wording is empirically hardened against real evasions rather than guessed; this is why the prose "sticks" where hand-written guidance doesn't.

**The SUBAGENT-STOP pattern.** Two related mechanisms share the name. (a) In the gateway, a guard: *"If you were dispatched as a subagent to execute a specific task, skip this skill"* — preventing recursive re-discovery in already-scoped subagents ([DeepWiki](https://deepwiki.com/obra/superpowers/7.1-using-superpowers-(meta-skill))). (b) In pressure-testing, the harness *stops* a subagent that failed to obey a skill and asks it *why*, surfacing the rationalization to feed back into the skill ([blog.fsck.com](https://blog.fsck.com/2025/10/09/superpowers/), [Heavybit podcast ep.36](https://www.heavybit.com/library/podcasts/open-source-ready/ep-36-managing-ai-coding-agents-with-jesse-vincent)). The unifying idea Vincent states plainly: *"Trying to get AI agents to 'understand the importance of TDD' is futile. Instead, create a system where they cannot proceed without adhering to skills"* ([dev.to/tumf](https://dev.to/tumf/superpowers-the-technology-to-persuade-ai-agents-why-psychological-principles-change-code-quality-2d2f)).

> **Flag (unverified/marketing):** GitHub-star counts cited across secondary sources are wildly inconsistent (50,000 / 93,000 / "150,000") and read as promotional; treat as unverified. The *structural* claims above are corroborated by primary source files and the author's own blog.

## 2. Prompt-engineering evidence on instruction-following / compliance

**Imperative/absolute vs soft framing.** Anthropic's guidance is to give "clear, direct, and detailed instructions" and to disambiguate any instruction "that can be interpreted in different ways" ([Anthropic prompt-engineering docs](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices)). Superpowers' empirical finding sharpens this: soft modal verbs ("prefer", "consider", "unless it matters") measurably "reopen negotiation" under pressure, whereas prohibitions plus pre-empted rationalizations hold ([writing-skills/SKILL.md](https://github.com/obra/superpowers/blob/main/skills/writing-skills/SKILL.md)). Caveat from Anthropic: keep the tone "neutral, professional"; a hostile/coercive tone "can lead to unpredictable results" — so *firm and absolute* ≠ *abusive* ([Anthropic best practices](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices)).

**Placement / recency — "lost in the middle."** The canonical result: models "achieve highest accuracy when relevant information appears at the beginning or end of the input context," and performance "degrades significantly when critical information is positioned in the middle" ([Liu et al., *Lost in the Middle*, TACL](https://direct.mit.edu/tacl/article/doi/10.1162/tacl_a_00638/119630/Lost-in-the-Middle-How-Language-Models-Use-Long)). Mechanistically this is attention/primacy bias at the start plus recency bias at the end, with RoPE distance-decay starving the middle ([survey via arXiv:2510.10276](https://arxiv.org/pdf/2510.10276)). **Direct implication:** a long agent system prompt that *embeds* a full workflow buries that workflow in the low-attention middle. The recommended mitigation is to put the most important instruction/context first and the query last — or, better, not to keep the procedure resident at all.

**Just-in-time injection vs static context (this is the crux).** Anthropic's [*Effective context engineering for AI agents*](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) argues context is "a finite resource with diminishing marginal returns" and that models exhibit **"context rot"** — degradation as token volume grows. It explicitly recommends **just-in-time retrieval** over front-loading: agents should "maintain lightweight identifiers … and use these references to dynamically load data into context at runtime using tools," analogous to humans using file systems/bookmarks rather than memorizing everything. It also endorses sub-agents that "handle focused tasks with clean context windows." This is precisely the superpowers pattern: a tiny gateway resident always, the heavy procedure loaded on invocation. *Embedding the procedure statically is the exact anti-pattern this guidance warns against.*

**Forcing announce/commit before acting.** Superpowers requires the agent to *announce which skill it is using* before proceeding; the third-party analysis attributes this to a commitment/consistency effect — "declaring skill use before proceeding is more effective than embedded prose … forces deliberate choice over drift" ([dev.to/tumf](https://dev.to/tumf/superpowers-the-technology-to-persuade-ai-agents-why-psychological-principles-change-code-quality-2d2f)). This aligns with a broader research line on making agents *verify/commit before acting*: "Verify Before You Commit" / self-auditing enforces checks "over internal belief states before action commitment" ([arXiv:2604.08401](https://arxiv.org/abs/2604.08401)), and work on **premature commitment** shows agents that "settle on one interpretation early and then defend it" is a measurable, addressable failure mode ([arXiv:2606.22936](https://arxiv.org/pdf/2606.22936)). The general principle: an explicit, externalized commitment step reduces silent drift.

**Decompose into checklists/todos and decision trees.** Consistent with Anthropic's "right altitude" guidance — avoid both brittle hardcoded logic and vague hand-waving, aiming for "strong heuristics" ([effective context engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)) — checklists convert an obligation into discrete verifiable states and decision trees force explicit branch selection at drift points (§1).

**Why embedded prose gets skimmed but an invoked action gets executed.** Synthesizing the above: (a) embedded prose sits in the low-attention middle of a long prompt (lost-in-the-middle); (b) it competes with everything else for a finite attention budget (context rot); (c) it presents *no action the model must take*, so there is nothing to commit to; (d) it is read once at prompt-build time and never re-surfaced. A discrete invoked skill inverts all four: it is loaded fresh (top-of-context recency), in a clean/focused window, as a required action the agent announces, exactly when relevant. **This is the mechanistic explanation for why moving superpowers' processes into agent prompts caused agents to stop following them.**

## 3. Enforcement mechanisms in real agent frameworks

The key distinction is **SOFT enforcement** (linguistic/prompt pressure — can always be rationalized past) vs **HARD enforcement** (the runtime intercepts and *denies* a non-conforming tool call — cannot be argued with).

**Claude Code hooks (the reference implementation of hard enforcement).** Hooks are "the deterministic layer around your agent" ([Claude Code hooks reference](https://code.claude.com/docs/en/hooks)). Which lifecycle events can actually **block**:

| Event | Can block? | Mechanism |
|---|---|---|
| `PreToolUse` | **Yes** | `permissionDecision: "deny"` (or exit code 2) |
| `PermissionRequest` | **Yes** | `decision.behavior: "deny"` / exit 2 |
| `UserPromptSubmit` | **Yes** | `decision: "block"` / exit 2 |
| `Stop` / `SubagentStop` | **Yes** | `decision: "block"` / exit 2 |
| `PostToolUse` | **No** | tool already ran; can only add `additionalContext` |

The deny reason (`permissionDecisionReason`) is **fed back to the model** so it can adapt ([Claude Code hooks reference](https://code.claude.com/docs/en/hooks)). Critically, a `PreToolUse` deny "is evaluated before any permission-mode check, so it blocks the tool even under `bypassPermissions`" — the user cannot escape it by switching modes ([hooks guide](https://claudefa.st/blog/tools/hooks/hooks-guide)). This is how you build the canonical hard gates: **block an `Edit` on source until a failing test file exists**, or **block a `Stop`/"done" claim until a verification command has run** — the exact "cannot proceed without adhering" property Vincent describes, but enforced by the runtime rather than by prose.

**Claude Code skills & subagents.** Skills are on-demand loaded `SKILL.md` files (soft, but with just-in-time loading); subagents run in clean context windows. Neither is hard on its own; they pair with hooks for hard gates.

**Cursor rules.** `.mdc` files with three frontmatter fields — `description`, `globs`, `alwaysApply`. `alwaysApply: true` injects the rule "into every prompt regardless of relevance" (a soft always-on layer, the analogue of the superpowers gateway) ([Cursor Rules docs](https://cursor.com/docs/rules)). Notable footgun: plain `.md` files in `.cursor/rules/` are "ignored silently" — only `.mdc` fires ([dev.to](https://dev.to/nedcodes/cursor-agent-mode-ignores-cursorrules-use-mdc-instead-5flb)). Team "Enforce this rule" removes the opt-out but is still prompt-level (soft).

**Cline / Roo.** `.clinerules/` directory of `.md`/`.txt` files combined into a unified rule set, with path-scoped rules ([search synthesis](https://cursor.com/docs/rules)). Soft, prompt-level.

**Policy / guardrail layers (research).** A body of work builds *runtime enforcement* that validates each proposed action against a policy before execution: **AgentSpec** does "deterministic enforcement at discrete execution checkpoints … before invoking high-impact API calls or committing irreversible actions," with strategies like `llm_self_examine` to let the agent recover ([arXiv:2503.18666](https://arxiv.org/pdf/2503.18666)); **VeriGuard** is "a runtime monitor to validate each proposed agent action against the pre-verified policy before execution," moving from empirical to formal verification ([arXiv:2510.05156](https://arxiv.org/html/2510.05156v1)). These are the academic generalization of Claude Code's `PreToolUse` deny.

**Tradeoffs & failure modes of hard gating.**
- *Brittleness / false blocks:* a gate keyed on a heuristic (e.g., "a test file must exist") blocks legitimate edge cases; mitigate by feeding a clear deny reason so the agent can satisfy the gate rather than fight it.
- *Bypass via delegation:* if the gate lives at one layer, agents route around it at another (see OpenCode #5894 in §4 — subagent tool calls skip plugin hooks). **Best practice: enforce at the lowest common layer** (runtime permission, not per-agent prose).
- *Adversarial drift under pressure:* soft-only enforcement predictably fails the sunk-cost/time-pressure scenarios superpowers documents; hard gates are immune to rationalization.
- *Over-gating kills flow:* too many `ask`/`deny` prompts train users to reflexively approve. Reserve hard gates for the few irreversible/high-value invariants (tests-before-code, verification-before-done); use soft structure for everything else.
**General principle:** soft enforcement raises the *probability* of compliance; hard enforcement makes the *violating path unrepresentable*. Robust systems use soft structure to guide the common case and hard gates to backstop the few invariants that must never break.

## 4. OpenCode.ai capabilities (verified against docs + GitHub)

Sources: [opencode.ai/docs/plugins](https://opencode.ai/docs/plugins/), [/docs/agents](https://opencode.ai/docs/agents/), [/docs/permissions](https://opencode.ai/docs/permissions/), [/docs/config](https://opencode.ai/docs/config/), community hook references ([johnlindquist gist](https://gist.github.com/johnlindquist/0adf1032b4e84942f3e1050aba3c5e4a)), and GitHub issues. **Note:** the repo now lives under the `anomalyco/opencode` org (historically `sst/opencode`); issue links use `anomalyco`.

**(a) Plugin hook API — can a hook DENY/abort a tool call?** **Yes, partially.** `tool.execute.before(input, output)` fires before a tool runs and **can block execution by throwing an error** (`throw new Error("…")`); it can also modify `output.args` ([plugins docs](https://opencode.ai/docs/plugins/), [johnlindquist gist](https://gist.github.com/johnlindquist/0adf1032b4e84942f3e1050aba3c5e4a)). `tool.execute.after` is read-only/observational (the tool already ran — cannot block). A `stop` hook can "intercept agent stop attempts" and re-engage the agent (the analogue of `SubagentStop`). An `event` hook (e.g. `session.idle`) is read-only. **Critical limitation (verified):** [Issue #5894](https://github.com/anomalyco/opencode/issues/5894) — confirmed & reproducible, now closed — plugin `tool.execute.before` hooks **successfully block primary-agent tool calls but do NOT intercept tool calls made by subagents spawned via the `task` tool**: *"Any agent can bypass plugin restrictions by delegating work to a subagent."* So **plugin-hook deny is not a reliable hard-enforcement layer in a multi-agent setup.**

**(b) `experimental.chat.system.transform`.** Documented as an async hook that receives `{ sessionID?, model }` and mutates an `output.system` string array to "inject context into the system prompt," running before chat processing ([johnlindquist gist](https://gist.github.com/johnlindquist/0adf1032b4e84942f3e1050aba3c5e4a)). **It is experimental and was reported broken:** [Issue #17100](https://github.com/anomalyco/opencode/issues/17100) — confirmed by two independent plugins — the plugin mutates `output.system` in-process (unit tests pass) but "the Go runtime silently discards all mutations before the system prompt reaches the LLM." Status: **closed as "not planned"** (labels `core`, `needs:compliance`); **no workaround identified.** A related request ([#17637](https://github.com/anomalyco/opencode/issues/17637)) asks to include user-message text in the hook input. **Ordering/multiple-plugin registration is undocumented — flag as uncertain.** Notably, the superpowers OpenCode adapter injects its bootstrap via **`experimental.chat.messages.transform`** (a *messages* transform), not `system.transform` ([superpowers README.opencode.md](https://github.com/obra/superpowers/blob/main/docs/README.opencode.md)) — suggesting `messages.transform` is the working injection path in practice. **Recommendation: prefer `messages.transform`; do not rely on `system.transform` given #17100.**

**(c) Force a DEFAULT / primary agent.** **Yes.** `opencode.json` supports **`default_agent`**, which "determines which agent is used when none is explicitly specified"; it must be a *primary* agent (built-in `build`/`plan` or a custom one); if it's missing or a subagent, OpenCode "falls back to `build` with a warning." Applies across TUI, `opencode run` (CLI), desktop, and GitHub Action ([config docs](https://opencode.ai/docs/config/)). **This means you can force all sessions into your orchestrator primary agent by default.** (Users can still switch primaries with Tab / `@name`.)

**(d) How skills are invoked; do subagents inherit injected system-prompt content?** Skills are invoked via OpenCode's **native `skill` tool** (there is a `skill` permission key) ([permissions docs](https://opencode.ai/docs/permissions/)); superpowers maps "invoke a skill" → OpenCode's native skill tool ([README.opencode.md](https://github.com/obra/superpowers/blob/main/docs/README.opencode.md)). Subagents are dispatched via the **`task` tool** with a `subagent_type` (e.g. `general`, `explore`). Whether a subagent inherits primary-session injected system content is **not clearly documented**; the practical evidence is mixed and points to **subagents starting with their own (clean) context** — hence #5894 (plugin hooks don't reach subagent tool calls) and the superpowers SUBAGENT-STOP guard that assumes subagents are separately scoped. **Flag as partially uncertain — treat subagents as fresh contexts that must be re-injected/re-briefed, not as inheritors.**

**(e) Agent frontmatter capabilities.** Markdown agents at `<name>.md` (filename = agent name); **the body becomes the system prompt**, frontmatter is metadata ([agents docs](https://opencode.ai/docs/agents/)). Fields: `description` (required), `mode` (`primary` | `subagent` | `all`, default `all`), `model`, `temperature`, `top_p`, `steps` (max agentic iterations), `disable` (bool), `color`, `tools` (**deprecated** — prefer `permission`), and `permission`. `permission` maps tools → `allow` | `ask` | `deny` with glob patterns and **last-matching-rule-wins**; `permission.task` restricts which subagents an agent may invoke. `tools: true`≡`{"*":"allow"}`, `false`≡`{"*":"deny"}`.

**(f) Native rules / AGENTS.md.** **Yes** — OpenCode has a native rules mechanism ([rules docs](https://opencode.ai/docs/rules/)) and reads `AGENTS.md` as an always-on context file. This is a *soft* always-on layer (prompt-level), the OpenCode analogue of Cursor's `alwaysApply` / the superpowers gateway — but as prose it is subject to the same lost-in-the-middle drift; it is not hard enforcement.

**(g) Native permission maps = OpenCode's real hard-enforcement layer.** `permission` in `opencode.json` (global) and per-agent (`agent.<name>.permission`) gates the tools `read, edit (edit/write/patch), glob, grep, bash, task, skill, lsp, question, webfetch, websearch, external_directory, doom_loop` with `allow`/`ask`/`deny`, globbed, last-match-wins ([permissions docs](https://opencode.ai/docs/permissions/)). These are described as **runtime-enforced hard constraints**, and the docs state subagents "don't bypass global permissions — agent-level rules override global ones." **Important nuance:** this contradicts the plugin-hook bug (#5894) only in appearance — they are *different layers*. Native `permission` deny is enforced by the runtime (and per docs reaches subagents); **plugin `tool.execute.before` deny is not reached by subagents.** So for hard gates on OpenCode, **use native `permission` maps, not plugin throws.** (Edge-case caveat: [Issue #6396](https://github.com/anomalyco/opencode/issues/6396) reports per-agent `deny` in `opencode.json` being ignored when the agent is invoked via the **SDK** — verify against your invocation path.)

## 5. Synthesis — a compliance pattern language

Patterns ranked by **leverage × evidence strength**, tagged **SOFT** (portable, prompt-level) / **HARD** (needs platform hooks), and **portable** vs **OpenCode-specific**.

**Tier 1 — highest leverage (do these first):**

1. **Always-on gateway that forces a skill check *before any response*.** *(SOFT; portable.)* The single most load-bearing superpowers element. Re-inject a tiny bootstrap every turn carrying the 1% Iron Law + rationalization table. Evidence: strong (primary source + lost-in-the-middle explains why "resident tiny + loaded-on-demand" beats "embedded once"). *OpenCode:* inject via `experimental.chat.messages.transform` (the path superpowers actually uses) and/or `AGENTS.md`; **avoid `system.transform`** (#17100).
2. **Keep every procedure a discrete, invokable skill — never embedded prose.** *(SOFT; portable.)* This is the direct fix for the regression this repo hit. Just-in-time loading (Anthropic context-engineering) + top-of-context recency + a required, announceable action. Evidence: strong.
3. **Hard-gate the 2–3 invariants that must never break** (e.g. no source edit before a failing test; no "done"/Stop without a verification command). *(HARD.)* Evidence: strong (Claude Code `PreToolUse` deny; AgentSpec/VeriGuard). *Claude Code:* `PreToolUse` + `Stop`/`SubagentStop` deny. *OpenCode:* native `permission` maps (runtime-enforced, reach subagents) — **not** plugin `tool.execute.before` (bypassed by subagents, #5894). *Portable across runtimes:* the *idea* is portable; the *mechanism* differs per platform.
4. **Force sessions into the orchestrator primary agent by default.** *(HARD-ish config; OpenCode-specific.)* `default_agent` in `opencode.json`. Guarantees the workflow scaffolding is present without the user opting in.

**Tier 2 — wording & structure that make SOFT enforcement actually hold:**

5. **Iron Law + rationalization table + red-flag list** for any discipline skill. *(SOFT; portable.)* Pre-empt the agent's own most-likely excuses in-context.
6. **"Match the form to the failure" + "no nuance clauses."** *(SOFT; portable.)* Prohibitions for rule-violations; positive recipes for output-shape; REQUIRED-field templates for missing elements; observable predicates for conditionals. Never append "unless it matters."
7. **Announce/commit before acting** — require the agent to declare the skill/step it is entering. *(SOFT; portable.)* Commitment-consistency + "verify before commit" research.
8. **Checklists → real todos**, one per gate item. *(SOFT; portable.)* Externalizes state so steps can't be silently skipped.
9. **"Use when…" trigger-only descriptions** that never leak the workflow. *(SOFT; portable.)* Prevents the summary-shortcut.
10. **Decision flowcharts only at genuine branch/stop points.** *(SOFT; portable.)*

**Tier 3 — process/meta:**

11. **Pressure-test the wording (TDD-for-prompts):** run the skill against fresh agents under time/sunk-cost/authority pressure *with a no-guidance control*, capture verbatim rationalizations, add counters, repeat. *(SOFT process; portable.)* This is *how* Tier-2 wording gets hardened; without it you are guessing.
12. **Treat subagents as fresh contexts** — re-brief/re-inject; never assume they inherit the primary's injected guidance. *(Portable caution; acute on OpenCode per #5894 and clean-context subagents.)*

**Quick SOFT vs HARD map:** patterns 1, 2, 5–12 are prompt-level and portable across Claude Code / OpenCode / Codex-style runtimes. Patterns 3 and 4 need platform hooks and are where OpenCode diverges from Claude Code — on OpenCode, lean on **native `permission` maps + `default_agent` + `messages.transform` injection**, and do **not** depend on `tool.execute.before` deny or `system.transform` mutation.

---

## Recommendations for this redesign

The regression has a clean diagnosis: **embedding the design→plan→build→review workflow into agent system prompts deleted the three mechanisms that made it stick — the forced pre-response invocation, the just-in-time load, and the announce/commit step — and buried the guidance in the low-attention middle of a long prompt.** Concretely, for this OpenCode framework:

1. **Restore an always-on gateway.** Re-establish a `using-superpowers`-style bootstrap injected every turn via the plugin's `experimental.chat.messages.transform` hook (the path the superpowers OpenCode adapter uses), carrying the 1% Iron Law and a rationalization table that forces a skill/step check *before any response*. Do **not** rely on `experimental.chat.system.transform` — it was reported to silently discard mutations and closed "not planned" ([#17100](https://github.com/anomalyco/opencode/issues/17100)). Verify your injection reaches the model with a marker probe before trusting it.
2. **Re-decompose the embedded workflow back into discrete invokable skills** (brainstorm, writing-plans, executing-plans/subagent-driven-development, requesting/receiving-code-review, verification-before-completion). The orchestrator's job is to *invoke and announce* each, not to inline the procedure. This is the primary fix.
3. **Set `default_agent` to your orchestrator primary** in `opencode.json` so every session starts inside the scaffolding.
4. **Add native-`permission` hard gates for your 2–3 non-negotiables** — e.g. `build`-subagent `edit`/`bash` restrictions, and consider gating "completion" behind a verification step. Use the runtime `permission` map (reaches subagents), **not** plugin `tool.execute.before` (bypassed by subagents, [#5894](https://github.com/anomalyco/opencode/issues/5894)). If you invoke agents via the SDK, verify per-agent `deny` is honored ([#6396](https://github.com/anomalyco/opencode/issues/6396)).
5. **Re-brief subagents explicitly** — pass the relevant skill/brief into each `task` dispatch rather than assuming inheritance; treat every subagent as a clean context.
6. **Rewrite each discipline skill to the superpowers form** — Iron Law, rationalization table, red-flag list, "match the form to the failure," no nuance clauses — and **pressure-test** each against a fresh agent under time/sunk-cost pressure with a no-guidance control before shipping.
7. **Keep skills tight** (<200 words for always-loaded, <500 otherwise) to protect the attention budget — consistent both with Anthropic's context-rot guidance and this repo's existing CLAUDE.md targets.

**Net:** soft structure (gateway + discrete skills + hardened wording + announce/commit) fixes ~80% of the drift and is fully portable; a thin layer of hard gates via native OpenCode `permission` maps backstops the few invariants that must never break. Avoid the two OpenCode traps (`system.transform` discard; plugin-hook subagent bypass) by routing injection through `messages.transform` and enforcement through native permissions.
