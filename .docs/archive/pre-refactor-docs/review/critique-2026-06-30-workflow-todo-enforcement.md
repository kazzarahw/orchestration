## Critique Report: Workflow & Todo Enforcement Design Spec

### Context
Design spec review (pre-implementation). Reviewing the spec document at `src/docs/plans/design-2026-06-30-workflow-todo-enforcement.md` before any planning or coding begins.

### Severity Summary
- Critical: 3 — Must fix before proceeding
- High: 5 — Should fix, will cause problems later
- Medium: 4 — Worth fixing, notable but not urgent
- Low: 3 — Style/preference, optional
- Info: 2 — Observations, not actionable

### Critical Issues

1. **Tool mapping migration into SKILL.md breaks multi-platform design** — `src/docs/plans/design-...md` §"Component 1" + §"File Changes Summary" (line 148)
   - **Problem:** The spec says the OpenCode-specific tool mapping (currently generated by `superpowers.js` at injection time) should be editorially embedded into `using-superpowers/SKILL.md` as a `+~15 lines` edit. However, `using-superpowers` is explicitly designed as a **multi-platform** skill — it has per-platform reference files under `references/` (claude-code-tools.md, codex-tools.md, copilot-tools.md, gemini-tools.md, pi-tools.md, antigravity-tools.md) and its body says "Skills speak in actions... For per-platform tool equivalents and instructions-file conventions, see [references/]". Embedding an OpenCode-specific tool mapping directly into SKILL.md makes the skill **OpenCode-specific**, which contradicts its multi-platform mandate. The tool mapping is currently injected by `superpowers.js` only on OpenCode — that's correct behavior. Moving it into the skill itself means every platform gets OpenCode-specific tool names, breaking Claude Code, Codex, Gemini, etc.
   - **Impact:** The `using-superpowers` skill becomes unportable. Other platforms see OpenCode-specific tool names (`todowrite`, `apply_patch`) in the skill body, creating confusion. The multi-platform reference system becomes dead code.
   - **Suggestion:** Keep the tool mapping in the plugin (like `superpowers.js` does). The plugin knows which platform it's running on. Alternatively, create a platform-specific injection wrapper in the plugin that embeds the correct tool mapping per-platform. Don't put platform-specific content into a cross-platform skill document.

2. **Config key `skill_injection` risks collision with OpenCode namespace** — `src/docs/plans/design-...md` §"Component 1", line 51-56
   - **Problem:** The spec introduces a top-level OpenCode config key `"skill_injection"`. OpenCode's config schema (`opencode.json`) is an evolving API. Using an unnamespaced top-level key risks collision with future OpenCode config properties. If OpenCode later adds a native `skill_injection` feature with different semantics, your plugin's config and OpenCode's native config will conflict silently.
   - **Impact:** Undefined behavior if OpenCode adopts a `skill_injection` key. Either one overrides the other, or they merge unpredictably. Could break on OpenCode update.
   - **Suggestion:** Namespace the config key under the plugin name, e.g., `"plugins": { "skill-injection": { "skills": [...] } }` or at minimum `"skill_injection_skills"`. The current `superpowers.js` doesn't need config because it's hardcoded — the new plugin is config-driven by design, so proper namespacing is essential.

3. **Skills path registration is underspecified in transition** — `src/docs/plans/design-...md` §"Component 1", line 65
   - **Problem:** The spec's table says the `config` hook should "Register skills path (from superpowers.js). Read `skill_injection` array from config." but provides zero detail on **how** the plugin determines the skills directory path. The current `superpowers.js` uses `path.resolve(__dirname, '../skills')` — a path relative to the plugin file. The spec doesn't specify whether the new plugin follows the same pattern, reads from config, or uses a different mechanism. This is a critical implementation detail that's entirely unspecified.
   - **Impact:** The new plugin cannot be implemented from this spec alone. The path discovery mechanism must be specified.
   - **Suggestion:** Add explicit documentation of path resolution. If staying with `__dirname`-relative resolution (simplest, keeps install.sh compatibility), state that explicitly. If moving to config-driven paths, define the config schema.

### High Issues

1. **No per-skill analysis of subagent injection appropriateness** — `src/docs/plans/design-...md` §"Architecture", lines 31-34
   - **Problem:** The spec says all three injected skills (`using-superpowers`, `using-todos`, `maximizing-information-density`) go to "Every agent... (primary + subagent, no session gating)." But `maximizing-information-density` instructs agents to apply lossy compression techniques to human-facing output. Subagents like `implement.md` produce structured reports with specific required formats (Status, Commits, Test summary). If a subagent applies `maximizing-information-density` to its report, it might compress away required fields, making the report unparseable by the orchestrator. The skill itself says "Agent audience → lossless only" but it's a long skill and the agent might not correctly self-classify.
   - **Suggestion:** Either (a) exclude `maximizing-information-density` from injection into subagents, or (b) explicitly verify its audience-determination rubric prevents subagents from applying lossy techniques to structured reports, and document this analysis in the spec.

2. **develop.md 342→100 line reduction without mapping where detail migrates** — `src/docs/plans/design-...md` §"Component 3", lines 107-116
   - **Problem:** The spec says phase descriptions go from "20-50 lines each with sub-steps" to "1-2 lines: skill reference + gate condition." This is a 70% reduction. The spec doesn't map where each lost detail goes — is it absorbed by skills? Which skills? For example, the current develop.md has detailed error handling (15+ lines), stopping conditions (5+ lines), cross-cutting rules (40+ lines), and subagent dispatch rules (20+ lines). The spec says "Bulk discipline comes from injected skill" (referring to `using-todos`), but `using-todos` only covers todo discipline, not error handling, stopping conditions, or the detailed review-loop mechanics.
   - **Suggestion:** Create a mapping table: for each phase/section removed from develop.md, state which skill or mechanism provides coverage. If some detail is genuinely removed (not migrated), call that out explicitly as a scope reduction. Without this, the 100-line prompt may produce an orchestrator that doesn't know how to handle subagent BLOCKED status, review loops, or escalation paths.

3. **Temperature change (0.2 → 0.1) for develop agent lacks rationale** — `src/docs/plans/design-...md` §"Component 3", line 112
   - **Problem:** The spec changes the develop agent's temperature from 0.2 to 0.1 with no justification. For an orchestrator that dispatches subagents and needs to interpret ambiguous subagent responses, 0.1 may be too deterministic — the agent may fail to recognize edge cases and blindly follow its (now greatly reduced) prompt. Conversely, 0.2 was already quite low. Changing temperature is a behavioral tuning parameter, not a design constraint embedded in a spec.
   - **Suggestion:** Either remove the temperature change from the spec (leave at 0.2) or add a clear rationale. If 0.1 is intended to reduce "creative" interpretations of the lifecycle, that should be stated and benchmarked (e.g., "observed behavior: at 0.2 the agent occasionally skips critique gate; 0.1 eliminates this").

4. **Frontmatter stripping loses skill identity for injected skills** — `src/docs/plans/design-...md` §"Injection Format", lines 73-75
   - **Problem:** The plugin strips YAML frontmatter (name, description) and wraps only the body in `<AUTO_INJECTED_SKILL>` markers. The `using-todos` skill's identity metadata is lost. If the agent would need to reference the skill by name (e.g., to invoke it via the `skill` tool), it can't — it only has the body. The `<AUTO_INJECTED_SKILL name="using-todos">` marker helps, but the description is gone. More importantly, the agent might waste tokens trying to `skill`-invoke a skill it already has injected.
   - **Suggestion:** Include key frontmatter fields (at minimum `name` and `description`) inside the `<AUTO_INJECTED_SKILL>` wrapper tag or as a comment at the top of the injected content. This gives the agent enough metadata to recognize it already has the skill and avoid redundant invocations.

5. **Subagent todo lists have no durability mechanism** — `src/docs/plans/design-...md` §"Out of Scope", lines 166-167, and §"using-todos Skill", lines 87-101
   - **Problem:** The spec says "No filesystem-based todo persistence — todowrite is the sole durable record." The `using-todos` skill says subagents get pre-populated todos. But if a subagent session compacts or the subagent loses context, its todo list is gone with no recovery. The SDD skill has a progress ledger for this (`progress.md` in `.opencode/sdd/` after migration), but the `using-todos` skill doesn't reference it. An implementer subagent dispatched by SDD has two tracking mechanisms (todos + ledger) with no bridge between them.
   - **Suggestion:** Either (a) have the `using-todos` skill reference the SDD progress ledger as a recovery mechanism, or (b) explicitly document that subagent todos are ephemeral and the SDD orchestrator is responsible for tracking. Don't leave a gap between the two systems.

### Medium Issues

1. **OpenCode config key `skill_injection` validation unknown** — `src/docs/plans/design-...md` §"Component 1", lines 49-57
   - **Problem:** OpenCode's config loading may validate against a schema and reject unknown top-level keys. The `superpowers.js` plugin currently doesn't read from config at all — it's fully self-contained. The new plugin introduces a dependency on config having a `skill_injection` property that OpenCode may silently strip or warn about.
   - **Suggestion:** Verify OpenCode config schema behavior. If unknown keys are stripped, document the config registration mechanism (some platforms allow extending the config schema via plugin hooks). Add a fallback: if `skill_injection` is not present in config, default to the current hardcoded set (`using-superpowers`, `using-todos`) so the ecosystem doesn't silently break.

2. **Triage UX change — direct user question without assessment** — `src/docs/plans/design-...md` §"Component 3", line 115
   - **Problem:** The spec changes triage from agent-assessed ("This looks small — skip full lifecycle? [y/N]") to a direct user question: "Full lifecycle, fast path, or maintenance?" This pushes workflow knowledge onto the user, who may not know what "fast path" means or whether their request qualifies. The current approach is more user-friendly: the agent assesses and asks for confirmation.
   - **Suggestion:** Keep the agent assessment step before asking. The agent can still classify the request and present options: "This looks like [full lifecycle/fast path/maintenance]. Proceed with [option]? [y/N]". The direct-question approach in the spec regresses UX.

3. **Empty `skill_injection` array silently breaks the skill ecosystem** — `src/docs/plans/design-...md` §"Edge Cases & Risks", line 177
   - **Problem:** The spec says empty array = no injection, and the plugin "handles gracefully." But `using-superpowers` is the foundational bootstrap skill — without it, agents don't know to load skills, don't know the skill flow diagram, and don't know the 1% rule. An empty or missing config array means the entire superpowers ecosystem is disabled with no warning. The old `superpowers.js` was always-active (no config needed).
   - **Suggestion:** Add a warning log (via `console.warn` or similar) when `skill_injection` is empty or undefined: "Warning: skill_injection is empty — no skills will be auto-injected. Agents will not receive superpowers bootstrapping." Consider also documenting the default set if config key is missing.

4. **SDD script shell quoting is fragile and should be fixed during migration** — `src/skills/subagent-driven-development/scripts/task-brief` line 24, `review-package` line 26
   - **Problem:** The scripts use `$("$(cd "$(dirname "$0")" && pwd)/sdd-workspace")` which runs `sdd-workspace` as a command. The quoting chain is fragile: if `dirname "$0"` produces a path with spaces, the outer quoting may break. The spec is touching these files anyway (path migration `.superpowers → .opencode`).
   - **Suggestion:** Fix the quoting during migration. Use `SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` then `"$SCRIPT_DIR/sdd-workspace"`. This is a well-known shell pattern that handles spaces correctly.

### Low Issues

1. **`<SUBAGENT-STOP>` block creates noise for injected subagents** — `src/skills/using-superpowers/SKILL.md` lines 6-8
   - **Problem:** The `using-superpowers` skill has a `<SUBAGENT-STOP>` block: "If you were dispatched as a subagent to execute a specific task, skip this skill." When this skill is injected into every subagent's first user message via the plugin, the agent sees a block telling it to skip the content it's already been given. The agent wastes tokens processing this instruction and may be confused ("should I follow this or ignore it since it was injected?").
   - **Suggestion:** When the plugin injects skills into subagents, strip the `<SUBAGENT-STOP>` block. The plugin knows whether it's injecting into a primary or subagent session (check session metadata), so it can conditionally strip that section.

2. **`plan.md` and `build.md` not updated despite being in the agent ecosystem** — `src/docs/plans/design-...md` §"Other Agents — Minor Updates", lines 118-126
   - **Problem:** The spec adds todo lines to 5 agents (implement, review, critique, research, dogfood) but doesn't mention `plan.md` or `build.md`, which exist in `src/agents/` (though marked `disable: true` per AGENTS.md). If these agents are ever re-enabled, they'll lack todo discipline.
   - **Suggestion:** Either update the disabled agents too (low effort, future-proofing) or explicitly note they're excluded because disabled.

3. **SDD progress ledger path not referenced in `using-todos` skill** — `src/docs/plans/design-...md` §"Component 2", lines 87-101
   - **Observation:** The `using-todos` skill says "Your todo list is the source of truth" but the SDD skill maintains a filesystem progress ledger at `.opencode/sdd/progress.md` for durability. These two truth sources are disconnected. A subagent using the `using-todos` skill might faithfully maintain its todo list, have it lost to compaction, and have no awareness of the progress ledger as a fallback.
   - **Suggestion:** Add one line to `using-todos` referencing the progress ledger: "For long-running work, check for a progress ledger at `.opencode/sdd/progress.md` — if it exists, reconcile your todos against it."

### Info

1. **`install.sh` currently copies plugins as-is** — `install.sh` lines 34-47
   - **Observation:** The install script copies `src/plugins/` to `~/.config/opencode/plugins/`. Deleting `superpowers.js` and adding `skill-injection.js` in `src/plugins/` will automatically propagate through install.sh. No changes needed to the install script itself. This is correct.

2. **SDD skill references `@review.md` and `@critique.md` directly** — `src/skills/subagent-driven-development/SKILL.md` lines 268-269, 410-411
   - **Observation:** The SDD skill references these agents by filename. The spec correctly adds todo lines to both, but the SDD skill itself may need a `using-todos` reference in its workflow text (currently it says "create todos" in the flow diagram but doesn't formally load the skill). This is a minor consistency gap that doesn't block the spec but could be addressed.

### Positive Notes

- **Config-driven injection is a clear architectural improvement** over hardcoded skill references in `superpowers.js`. No code changes needed to add/remove injected skills.
- **Flat array config** — no nesting, no per-agent overrides, no session-type gating. Simple is the right choice.
- **`messages.transform` (one-time) instead of `system.transform` (per-round)** — correct cost optimization. Skill content is static; re-injecting every round wastes tokens.
- **SDD namespace migration (`.superpowers/` → `.opencode/`)** is overdue cleanup. The `.superpowers/` name is a legacy artifact from the Superpowers era. `.opencode/` is the correct namespace.
- **Marker-based double-injection guard** — simple and effective. The `<EXTREMELY_IMPORTANT>` check is replaced by `<AUTO_INJECTED_SKILL>` which is more specific and less likely to false-positive.
- **Risk table** in the spec (lines 172-179) is good practice. The identified risks are real and the mitigations are reasonable (though see Medium issue #3 about empty-array silence).

### Overall Assessment

**Not ready to proceed.** The spec has three critical issues that must be resolved: (1) the tool mapping migration into SKILL.md destroys the multi-platform design of the `using-superpowers` skill, (2) the unnamespaced config key risks collision with OpenCode's own config, and (3) the skills path registration mechanism is underspecified to the point where the plugin can't be implemented. Beyond these, there are five high-severity issues around subagent injection appropriateness, develop.md detail mapping, temperature rationale, frontmatter identity loss, and todo durability gaps. The core idea (config-driven skill injection + todo enforcement) is sound, but the execution details around platform compatibility and prompt reduction need thorough rework. Recommend revising the spec to address critical and high issues before proceeding to planning.
